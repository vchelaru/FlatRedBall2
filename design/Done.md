## Ground-Snap Slope Gate + Default Cut to Half-Tile
`PlatformerBehavior.ConsiderSnappingTo` now skips snap when `CurrentSlope == 0` (prior-frame surface was flat), emitting a new `skip: prior surface flat` diagnostic. Walking off a flat ledge onto lower flat ground — the staircase case — falls ballistically rather than teleporting onto the next step. Snap remains active for hugging downslopes across tile seams (the feature's actual purpose: `CurrentSlope != 0` from last frame's slope probe passes the gate). `PlatformerValues.SlopeSnapDistance` default lowered from `16f` → `8f`; the historical 16 matched a one-tile step exactly and was aggressive enough to visibly teleport even on staircase geometry where the slope gate would've caught it anyway. Regression surfaced in `AutoEvalCoinHopperSample` — walking right off a stepped ledge snapped to the next step; walking left off a tall ledge fell normally (asymmetric because only the short drop was in snap range). 7 existing snap tests were unwittingly exercising the old flat-to-flat behavior and needed `player.Behavior.CurrentSlope = 30f` priming to reach the gates they were named for — e.g. `GroundSnap_SurfaceTooSteep_DoesNotSnap` had been passing-for-the-wrong-reason; rewriting it now genuinely exercises the steepness gate. `GroundSnap_DownslopeToFlat_StaysGrounded` was the clearest case of a mis-named test (no downslope in the setup) — now primes slope and actually tests what the name implies. 4 new tests pin the new semantics: `GroundSnap_FlatToFlatWithGap_DoesNotSnap`, `GroundSnap_FlatToFlatWithGap_EmitsPriorSurfaceFlatDiagnostic`, `GroundSnap_SmallNonZeroSlope_StillSnaps` (gate is strictly non-zero, no implicit angle threshold — 5° qualifies), `DefaultSlopeSnapDistance_IsHalfTile`. `platformer-movement` skill, `content-boundary`-adjacent template JSON comment updated with the new gate rule.

## Platformer Jump Height — Airborne-Gravity Derivation
`SetJumpHeights(min, max?)` historically used `this.Gravity` for both the initial impulse and the sustain-length math. That's wrong when the slot it lives on (typically `ground`) has a different `Gravity` than the airborne slot that actually governs the jump trajectory — collision cancels ground gravity, so the jump arc runs entirely under airborne physics. Concrete case: `ground.Gravity = 4400`, `air.Gravity = 800`, `minJumpHeight = 24`, `maxJumpHeight = 84` produced `JumpVelocity ≈ 459.6` (tuned for g=4400) but the entity actually flew under g=800, reaching ~192 units (12 tiles) on full hold instead of the authored 84. New overload `SetJumpHeights(minHeight, maxHeight?, jumpGravity)` takes the trajectory gravity explicitly. `PlatformerConfigExtensions.ApplyTo` resolves `air.Gravity` once per call and passes it to every slot's derived-mode computation; fallback to the slot's own `Gravity` when no `air` slot is authored preserves ground-only configs without regression. `PlatformerValues.Gravity` XML now calls out that the field only acts while airborne — a ground slot's `Gravity` is effectively ceremonial for trajectory purposes, useful only as the fallback-gravity hint when no air slot exists. Per-env gravity (water, space) remains intentionally slot-based since gravity is an environment property; the rule "derived mode uses the paired airborne slot's gravity" generalizes to future (groundUnderWater, airUnderWater) pairs. 7 new tests in `PlatformerConfigTests` + `PlatformerValuesTests`: ground-derived with ground≠air uses air gravity for velocity *and* sustain length, air-derived uses own gravity, ground-only falls back to own gravity, raw mode unaffected, explicit-gravity overload non-positive throws. Template JSON + `platformer-movement` skill gotcha documented. AutoEvalCoinHopperSample hand-verified end-to-end.

## PNG Hot-Reload (Content Hot-Reload Increment 6)
`ContentManagerService.Load<T>(string path)` now routes on file extension: `Load<Texture2D>("Content/foo.png")` (extension present) loads the PNG directly from disk via `Texture2D.FromFile`, caches by normalized full path (case-insensitive), and tracks the resulting `Texture2D` in an internal registry; `Load<Texture2D>("foo")` (no extension) continues to delegate to MonoGame's compiled xnb pipeline, unchanged. Non-`Texture2D` `T`s also delegate to the xnb pipeline. The pair `Engine.Content.TryReload(string path) → bool` looks the path up in the registry, trial-loads the incoming file, and if dimensions match calls `existing.SetData(GetData(incoming))` so every live `Sprite` referencing the old `Texture2D` keeps working with no re-assignment. Dimension mismatch returns `false` (caller falls back to `RestartScreen(RestartMode.HotReload)`); unknown path also returns `false`. `UnloadAll` disposes and clears the registry alongside the existing tracked-disposables list. Internal seam for tests: `TextureLoader` + `TextureReloader` Funcs with production defaults that do `Texture2D.FromFile` + the `SetData` round-trip — tests substitute fakes so registry bookkeeping is covered without a real `GraphicsDevice` or disk I/O. `AnimationChainListSave.ToAnimationChainList` collapsed to a single overload taking `ContentManagerService` — routes frame textures through `Load<Texture2D>` with the path intact (extension preserved), so an .achx that references `"Player.png"` gets hot-reloadable frame textures for free; the old `GraphicsDevice` overload (which had its own local texture cache and bypassed the service) is removed, and `PlatformerSample.AnimatedPlayer` + `AutoEvalCoinHopperSample.Player/Coin` migrated. AutoEvalCoinHopperSample wired with a floating decorative `Bear.png` sprite (no gameplay behavior) and a `.png` branch in the `WatchContentDirectory` callback that dispatches to `Engine.Content.TryReload`; edits Bear.png → repaint appears mid-play without restarting. Scope limits documented in the `content-hot-reload` + `content-and-assets` skills: xnb-loaded textures aren't tracked (build artifact, can't reload at runtime), Aseprite atlas textures aren't tracked (generated in-memory, no single source PNG), TMX tileset textures aren't tracked (covered by the TMX restart fallback). 11 new tests in `ContentManagerServiceTextureTests` covering: TryReload-on-unknown-path, loader invocation + argument, caching (loader called once across repeat calls, including case-insensitive match), TryReload wires through to reloader with the tracked texture, TryReload propagates reloader's return, UnloadAll clears registry (subsequent TryReload false, subsequent Load re-invokes loader), and negative routing (non-Texture2D T / extension-less Texture2D path both skip the loader and hit the xnb pipeline). 549/549 green.

## TMX Hot-Reload (Content Hot-Reload Increment 5)
`TileMap.TryReloadFrom(string tmxPath)` re-parses a TMX file and applies tile-data changes in place: per-cell `TilemapTileLayer.SetTile` on the live layers (only on diff, so renderer churn is minimal) and clear+rebuild on every `TileShapeCollection` previously returned from `GenerateCollisionFromClass` / `GenerateCollisionFromProperty`. Returns `true` if applied, `false` if the new TMX differs structurally — caller falls back to `RestartScreen(RestartMode.HotReload)`. Structural compare covers map dims, `TileWidth/Height`, layer count + names + types, per-tile-layer dims, tileset count + name + `FirstGlobalId` + `TileCount`, object-layer object count (per-object equality is conservatively reduced to count diff for v1 — moving a spawn marker still triggers a restart). TSC tracking is automatic and quiet: `Generate*` records `(predicate, layerName, tsc)` internally so the caller's collision relationships keep working without re-attachment. Hand-authored mutations directly on a generated TSC (e.g. extra `AddPolygonTileAtCell` calls outside of `Generate*`) are wiped on in-place reload — those belong in `CustomInitialize` if they need to survive a full restart. New internal seam: `TileMap(Tilemap)` ctor + `TryReload(Tilemap)` so tests build hand-rolled `Tilemap` instances (matching `TileMapCollisionGeneratorTests` pattern) without GraphicsDevice or filesystem. Renderer cache refresh via `TilemapSpriteBatchRenderer.LoadTilemap(_tilemap)` post-reload (no-op in tests). 17 new tests in `TileMapReloadTests`: 9 structural-change cases (each → false), 4 in-place tile-data cases (no-op, add, remove, gid change), 2 polygon↔rect shape changes, 1 multiple-tracked-TSC case, 1 live-state-untouched-on-failure case. ## Content Hot-Reload: Editor Temp-File Filter
`Screen.CopyFileIfNeeded` now requires the destination file to already exist before copying source over it. Filters out editor noise: Photoshop scratch files (`~scratch.psb`), IDE autosaves, lock files, etc. that appear in the source content folder but were never copied to the build output by MSBuild. Without this filter, a directory-wide `WatchContentDirectory("Content", _ => RestartScreen(RestartMode.HotReload))` handler would fire on every transient temp file write — devastating for typical editor workflows. Trade-off: brand-new content files require one rebuild before the watcher picks them up (rebuild copies the file to dest, after which edits flow normally). Documented in the `content-hot-reload` skill. Existing integration tests updated to pre-create dest files (modeling prior MSBuild copy); 2 new tests pin the temp-file-skip behavior at both single-file and directory levels. 509/509 green.

## ContentWatcher: Source/Output Mapping + Directory Watch (Content Hot-Reload Increment 4)
Builds on the single-file `ContentWatcher` to handle the real two-folder problem (user edits source under `<csproj_dir>/Content/`, app reads from `bin/Debug/.../Content/`) and adds recursive directory watching. New on `FlatRedBallService`: `SourceContentRoot` (auto-detected at construction by walking up from `AppContext.BaseDirectory` for a `.csproj`; user-overridable; null in shipping builds with no csproj alongside the exe) and `OutputContentRoot` (defaults to `AppContext.BaseDirectory`). New on `Screen`: `WatchContent(string sourcePath, Action onChanged, string? destinationPath = null)` resolves source against `SourceContentRoot`, copies source → destination on each debounced change, then invokes the callback so user code that reads the standard output path sees fresh content. `destinationPath` defaults to `sourcePath`; override for non-default MSBuild `<TargetPath>` mappings. `WatchContentDirectory(string sourceDir, Action<string> onChanged, string? destinationDir = null)` watches a directory recursively; new `ContentDirectoryWatcher` uses **global debouncing** (waits until all writes across all files settle before processing the dirty batch — per-file debouncing would let earlier files reload while later files in the same burst were still being written), then for each dirty file copies and invokes callback (relative path passed in). `IDirectoryWatcher` interface + `FileSystemDirectoryWatcher` production wrapper around recursive `FileSystemWatcher` (filters out directory events to keep callbacks file-only). Both watcher types share the screen-scoped lifecycle from the previous increment — auto-disposed in `TeardownCurrentScreen`. `WatchContent`/`WatchContentDirectory` return `null` (no-op) when `SourceContentRoot` is null, so shipping builds skip hot-reload silently without any `#if DEBUG` gates. `ContentWatcher`'s copy delegate switched from `Action?` to `Func<bool>?` so a missing source file (deletion) cleanly skips the callback. 22 new tests across `ContentWatcherTests`, `ContentDirectoryWatcherTests`, `WatchContentIntegrationTests` (uses real temp directories for end-to-end copy verification), and `SourceContentRootTests` (csproj walk-up auto-detection). The `content-hot-reload` skill is rewritten to lead with directory watching, explain the two-folder model, and cover source/output customization.

## ContentWatcher Infrastructure (Content Hot-Reload Increment 3)
`Screen.WatchContent(string path, Action onChanged)` watches a file on disk and invokes the callback on the game thread (during `Update`) once writes settle. Returned `ContentWatcher` (also constructable directly with an `IFileWatcher` for testing) handles debouncing (150 ms default, configurable via `ContentWatcher.Debounce`), thread marshalling (file events fire on threadpool; `Tick(now)` runs on game thread), and `IOException` retry (file-mid-write triggers re-mark dirty, retried next debounce window). `IFileWatcher` is the testable abstraction; `FileSystemFileWatcher` is the production implementation wrapping `FileSystemWatcher` with `Changed`/`Created`/`Renamed` notification covering common editor save patterns. Watchers are screen-scoped: created via `Screen.WatchContent`, ticked once per frame by `FlatRedBallService.Update` (right after the pending screen change is flushed, before entity/collision/activity passes — so reloaded content lands in time for the rest of the frame), and auto-disposed in `TeardownCurrentScreen` so no per-screen cleanup is needed in user code. New skill `content-hot-reload` documents in-place reload (`Texture2D.SetData`, JSON value copy) vs `RestartScreen(RestartMode.HotReload)` (PNG dimension changes, TMX structural changes). 12 new tests: 8 unit tests on `ContentWatcher` covering the debounce window, multi-event collapse, IOException retry, post-callback re-firing, dispose detachment, and underlying-watcher event dispatch; 4 integration tests on `Screen` covering registration, per-frame ticking, and disposal on screen change / restart.

## Hot-Reload Restart Hooks (Content Hot-Reload Increment 2)
`Screen.RestartScreen(RestartMode mode)` overload + `RestartMode { DeathRetry, HotReload }` enum + `HotReloadState` typed key/value bag (`Set<T>` / `Get<T>` / `TryGet<T>`) + two virtual hooks on `Screen` (`SaveHotReloadState(HotReloadState)`, `RestoreHotReloadState(HotReloadState)`, both `public virtual` to match `CustomInitialize`/`CustomActivity`/`CustomDestroy` precedent). `RequestScreenRestart` now plumbs `RestartMode` through; the typed extension `RestartScreen<T>(this T, Action<T>, RestartMode = DeathRetry)` carries it too. On `RestartMode.HotReload`: engine constructs a fresh `HotReloadState`, calls `SaveHotReloadState` on the OLD instance pre-teardown (live state intact), runs the existing teardown → recreate → configure → `CustomInitialize` flow, then calls `RestoreHotReloadState` on the NEW instance — restore deliberately runs *after* `CustomInitialize` so saved values overwrite the freshly-built world (reverse order would let init clobber restore). `RestartMode.DeathRetry` never invokes the hooks, never constructs the bag — keeps the death/retry path bulletproof against accidentally preserving stale state across a death. 8 new tests in `ScreenTests` (3 bag, 5 hook ordering / round-trip / non-trigger / configure-interaction). Engine-managed defaults for camera + tracked-entity preservation are still open — covered by the next increment in the Content Hot-Reload roadmap.

## Screen Restart (Content Hot-Reload Increment 1)
`Screen.RestartScreen()` queues a deferred restart of the current screen at the start of the next frame: full teardown (entities, factories, content, Gum, async tasks via `_cts.Cancel()`) then a fresh `Activator.CreateInstance` of the same type. The engine retains a **single configure slot** (`Action<Screen>?` + `Type`); whoever set it last wins. `Start<T>(d)` and `MoveToScreen<T>(d)` set the slot. The typed extension `RestartScreen<T>(this T, Action<T>)` replaces the slot — both for that restart and for any subsequent plain `RestartScreen()`. Plain `RestartScreen()` replays whatever is currently in the slot. No property snapshot, no reflection over screen state — mid-game mutations to public properties are not silently captured. Common teardown extracted to `FlatRedBallService.TeardownCurrentScreen()` so `RequestScreenChange` and `RequestScreenRestart` share one path. Closure-staleness behavior is pinned by two probe tests (`RestartScreen_ClosureCapturedLocal_RestartReadsCurrentValueNotOriginal`, `RestartScreen_ClosureCapturedMutableReference_RestartReadsMutatedState`) — captured locals are read at restart time, not snapshot at `Start` time, which is documented as a footgun on `Start`/`MoveToScreen`/`RestartScreen` XML and in the `screens` skill (advice: pass literals, not captured locals). 11 tests in `ScreenTests`. Unblocks the rest of the Content Hot-Reload work (`HotReloadState`, engine-managed camera/tracked-entity restore, `ContentWatcher`).

## Aseprite (.ase / .aseprite) Runtime Loading
`FlatRedBall2.Content.Aseprite.AsepriteFileLoader.Load(path)` wraps `AsepriteDotNet` (1.9.1) to load an `.ase`/`.aseprite` file from disk. Two extension methods on `AsepriteFile` produce animation data. `ToAnimationChainList(GraphicsDevice)` processes the file into a packed atlas, creates a `Texture2D` in GPU memory, and returns an `AnimationChainList` with pixel-coordinate `SourceRectangle` per frame — each Aseprite tag becomes one `AnimationChain`, named after the tag. `ToAnimationChainListSave(string textureFileName)` does the same conversion without a graphics device and returns a save object with `CoordinateType = Pixel` — intended for offline `.achx` generation where the spritesheet PNG is exported separately. Untagged files fall back to a single chain named `AsepriteFileExtensions.UntaggedChainName` (`"Default"`) containing every frame in order, so users aren't confused when their first `.aseprite` has no tags. Frame durations are converted from Aseprite's `TimeSpan` to seconds. Unit tests cover the loader, tag → chain mapping, frame count, durations, texture-name propagation, pixel coordinate type, non-zero region dimensions, and the untagged fallback — using a custom binary `AsepriteTestFixture` that writes minimal valid `.ase` bytes so tests don't need fixture files on disk. Documented in the animation skill (`.claude/skills/animation/references/aseprite.md`); the skill's main SKILL.md now routes between `.ase` and `.achx` loading paths. `SlopesSample` uses this to load `soldier_girl.aseprite` on the player, flip between left/right-facing chains based on `PlatformerBehavior.DirectionFacing`.

## Overlay.TextScreen + Default Text Color/Z Fixes
`Overlay.Text(string, worldX, worldY)` follows world-space positions; the new `Overlay.TextScreen(string, screenX, screenY)` places a label at a fixed screen position in Gum canvas space (`TargetWidth`/`TargetHeight`) — preferred for HUDs, diagnostics, and any overlay that shouldn't move with the camera. Fixed two latent bugs in the Overlay label pipeline that produced unreadable text against the default `TextBackground`: (1) both `Text` and `TextScreen` now divide by `Camera.Zoom` when placing the Gum label, since Gum's canvas is viewport-pixels / zoom but `Camera.WorldToScreen` returns viewport pixels; `TextBackground` scales the stored canvas position back up by zoom before feeding `ScreenToWorld`. (2) `CreateLabel` now raises the label's `GumRenderable.Z` to `DefaultZ` (1000) so it sorts above its `TextBackground` rect at `DefaultZ - 1` (999) — previously the label's default Z of 0 sorted it *below* the background, and the semi-transparent black drew on top of the text. `CreateLabel` also explicitly sets the internal `TextRuntime.Color` to `White` so labels are legible on dark backgrounds out of the box; callers can recolor after `TextScreen`/`Text` returns. XML docs on `Text` and `TextScreen` now call out the world-vs-screen distinction up front to prevent the "why is my HUD text at the screen center" footgun.

## One-Way (Jump-Through / Cloud) Collision
`CollisionRelationship.OneWayDirection` (enum `None`/`Up`; `Down`/`Left`/`Right` throw `NotImplementedException`) gates a relationship so separation only fires when pushing the entity in the configured direction. For `Up`: three gates — `sep.Y > 0`, `VelocityY <= 0`, and a slope-aware `LastPosition` check. The separation's X component is zeroed so side-clipping entities are lifted straight up. `CollisionOccurred` does not fire on skipped pairs. `AllowDropThrough` (default `false`) decouples drop-through from the direction gate: `true` for cloud platforms (Down+Jump falls through), `false` for hard one-way barriers (Yoshi-style ratchet doors). Drop-through uses a one-frame suppression flag on `PlatformerBehavior` — after that frame, `LastPosition` is below the surface and the positional gate naturally prevents re-landing. Airborne Down-held provides continuous suppression for descending through stacked clouds. Gated by `PlatformerValues.CanFallThroughOneWayCollision` (default `true`). Sloped one-way tiles supported via `SlopeMode = PlatformerFloor` on the relationship — the `LastPosition` gate adjusts for the surface-Y delta between last-frame and current X positions. `SlopesSample` demonstrates flat and sloped cloud platforms. `AllowDropThrough` also suppresses ground-snap on `PlatformerFloor` relationships during active drop-through.

## Removed `UsesAcceleration` Flag (Platformer + TopDown)
Deleted the redundant `UsesAcceleration` bool from `PlatformerValues` and `TopDownValues`. In FRB1 the flag backed a "Instant vs Accelerated" radio button in Glue's editor; FRB2 has no editor, so it was a pure footgun — setting `AccelerationTimeX` without also flipping the bool silently produced instant movement. Now `AccelerationTimeX == Zero && DecelerationTimeX == Zero` is the sole "instant" signal. Migration: delete any `UsesAcceleration = ...` line; set the times to `Zero` if you want instant. All 16 samples, both test files, and both skill docs updated.

## Platformer Accel/Decel Reversal Fix
`PlatformerBehavior.Update` previously used `sign(diff) == sign(target)` to decide between `AccelerationTimeX` and `DecelerationTimeX`. This misclassified reversals (e.g. running right at +200 then pressing left → diff=-400, target=-200, both negative → "speeding up") and used accel-time to brake, making slippery configs feel instant-reversal. Now gated on `sign(velocity) == sign(target) && |velocity| < |target|`, matching FRB1: reversals, releases, and over-max states all brake at `DecelerationTimeX`; only genuine speed-up-in-target-direction uses `AccelerationTimeX`. 9 unit tests in `PlatformerBehaviorTests` pin the behavior across all eight velocity-vs-target quadrants.

## Slope Speed Adjustment (Platformer)
**Playtested in `SlopesSample` and confirmed working.** `PlatformerBehavior` scales `MaxSpeedX` by surface angle while walking on slopes. `CurrentSlope` (signed degrees, +X-rise positive) is refreshed each frame by a short downward raycast from any `PlatformerFloor` relationship (`ContributeSlopeProbe`), then consumed in `Update` to interpolate the effective max speed. Config on `PlatformerValues`: `UphillFullSpeedSlope`/`UphillStopSpeedSlope` (linear falloff to 0 between the two angles — at-or-above stop angle, input produces zero velocity) and `DownhillFullSpeedSlope`/`DownhillMaxSpeedSlope`/`DownhillMaxSpeedMultiplier` (linear boost from 1× toward the multiplier). Defaults match FRB1's predefined platformer profile (uphill 0→60°, downhill 0→60° with 1.5× boost) so any platformer with `PlatformerFloor` slope tiles gets the classic slope feel immediately; games without slope tiles never see non-zero `CurrentSlope` so the multipliers collapse to 1. Uphill/downhill detection compares the sign of input X against the sign of `CurrentSlope`; when airborne or on flat ground the multiplier collapses to 1. Accelerated and instant horizontal modes both use the adjusted speed (accel magnitude uses the effective cap when speeding up; decel keeps the raw `MaxSpeedX` so braking isn't slowed on an uphill).

## Platformer Ground Snap (Slope Adherence)
`PlatformerBehavior` probes downward after losing ground contact and snaps the entity to a nearby walkable surface so running off a downslope or off a short ledge doesn't briefly go airborne. Config on `PlatformerValues`: `SlopeSnapDistance` (default `16f`, set `0` to disable — the Sonic/ball-mode opt-out) and `SlopeSnapMaxAngleDegrees` (default `60f`, rejects walls). Enabled automatically by any `CollisionRelationship` whose `SlopeMode == PlatformerFloor` — no `SnapTarget` wiring needed; the player entity just implements `IPlatformerEntity { PlatformerBehavior Platformer { get; } }` and the relationship discovers it. `PlatformerBehavior.CollisionShape` (required when snap is active) supplies the feet position via `shape.AbsoluteY - shape.Height/2`; setting `SlopeSnapDistance > 0` with `CollisionShape == null` throws on the first probe attempt with a clear message. Multiple `PlatformerFloor` relationships all contribute snap candidates; `_snappedThisFrame` prevents double-snapping. `OnSnapDiagnostic: Action<string>?` emits per-frame reason codes (`[fN] snap: entityY=A→B probe=(...)→(...) hit=(...) cell=(col,row) shape=<kind> normal=(...)` / `[fN] skip: <reason>`) and `DiagLogPerFrameState` adds per-frame state traces — both allocation-free when unwired.

## SlopeMode on CollisionRelationship (not TileShapeCollection)
`SlopeCollisionMode` moved off `TileShapeCollection` and onto `CollisionRelationship` (default `Standard`). A single tile collection can now serve multiple relationships with different semantics (player as `PlatformerFloor`, ball/projectile as `Standard`). `TileShapeCollection.GetSeparationFor(shape, mode)` takes the mode per-call; relationship-less entry points (`CollidesWith`, public `GetSeparationVector`, `CollisionDispatcher`) default to `Standard`.

## Ceiling-Polygon Separation Fix (PlatformerFloor)
`TileShapeCollection` with `SlopeMode = PlatformerFloor` no longer teleports the player upward through a V-flipped slope tile (ceiling). `IsFloorLikePolygon` classifies each polygon by signed-area centroid Y in cell-local coords — negative centroid (mass in lower half) → floor-like, uses heightmap path; non-negative (mass in upper half) → ceiling-like, falls back to SAT (same path rectangles use). Prevents the old bug where the heightmap's "highest edge at centerX" rule pushed the player into the solid mass.

## Raycast Sub-Cell Rect and Polygon Start-Cell Support
`TileShapeCollection.Raycast` now traverses sub-cell rects in every entered cell (including the start cell), AND correctly handles the case where the ray starts inside a polygon's cell. Previously: (a) sub-cell rects were never tested during the DDA walk, so ground-snap over a curb/half-height tile missed; (b) the start cell's polygon was never tested at all — the DDA only looked at polygons on cell *entry* — so a ground-snap probe starting slightly below a slope's surface skipped the polygon entirely and hit whatever rect was beneath, producing a 16-unit teleport. Both paths merge hits by earliest ray-parameter. A new overload returns the hit shape: `Raycast(start, end, out hitPoint, out hitNormal, out ICollidable? hitShape)`; public helper `GetCellAt(Vector2 worldPoint)`. A ray origin inside a polygon's fill (even-odd point-in-polygon) returns an immediate hit at the start with normal = −ray direction ("you're already on the surface").

## Sub-Cell Rect Adjacency (RepositionDirections)
Sub-cell rects added via `AddRectangleTileAtCell` now participate in `RepositionDirections` seam suppression against three neighbor types: other sub-cell rects (both sides suppress at aligned edges), full-cell tiles (sub-cell rect face always suppressed; full-cell face suppressed only when the sub-cell rect **fully covers** the full-cell face — partial coverage leaves the full-cell face live for correct sideways push at exposed portions), and polygon tiles (sub-cell rect face suppressed when a polygon edge lying on the shared cell boundary fully covers the rect face's perpendicular extent). `AddPolygonTileAtCell` throws `InvalidOperationException` on duplicate `(col, row)` to surface multi-polygon authoring mistakes loudly rather than silently dropping. The polygon side of rect↔polygon seam suppression is deferred until the `Polygon.SuppressedEdges` SAT path is fixed (tracked separately).

`samples/SlopesSample/` is a side-scrolling platformer sample that validates TMX slope polygons (Phase 1), flip-flag handling, sub-cell rect collision, and ground snap end-to-end. Authored level uses the standard tileset's existing slope tile IDs (11/12/13) with Tiled H-flip for mirrored slopes — no custom slope tiles added. `IPlatformerEntity` is implemented on `Player`; `CollisionShape` wiring and `OnSnapDiagnostic → Debug.WriteLine` set up in `Player.CustomInitialize`.

## Slope Polygons in Tiled Maps — Phase 2 (sub-cell rectangles)
`TileMapCollisionGenerator` now honors plain `<object>` rectangle collision objects (`TilemapRectangleObject`) in addition to polygons. A tile with any custom collision object (polygon or rect) skips the default full-cell rect; both shape types emit side-by-side on the same tile if authored. Rects are converted from Tiled's Y-down top-left coords to FRB2 centered Y-up local offsets (`cx = x + w/2 - G/2`, `cy = G/2 - (y + h/2)`) and placed via the new `TileShapeCollection.AddRectangleTileAtCell(col, row, localCenterX, localCenterY, width, height)`. Multiple rects per cell are allowed (unlike polygons, which throw on duplicate). Sub-cell rects are stored in a separate `Dictionary<(col,row), List<AxisAlignedRectangle>>` so they participate in visual property propagation, `Clear`, `ShiftAllTiles`, and collision (`GetSeparationFor`) but not in `RepositionDirections` adjacency updates (they do not cover full cell faces). Query via `GetRectangleTilesAtCell(col, row)`. Tile flip flags (H, V, diagonal) are honored for rects: diagonal transposes both center (`(cx,cy) → (-cy,-cx)`) and size (swap w/h); H negates cx; V negates cy — applied D → H → V matching the polygon path. `TileShapeCollection.Raycast` does not yet test sub-cell rects (tracked in TODOS). The former `TryBuildPolygonPrototypes` helper in `TileMapCollisionGenerator` is now `BuildCollisionShapes`, producing both polygon prototypes and rect specs in a single pass.

## Slope Polygons in Tiled Maps — Phase 2 (flip flags)
`TileMapCollisionGenerator.TryBuildPolygonPrototypes` now honors `TilemapTile.FlipFlags`. Flips are applied in Tiled's declared order (diagonal → horizontal → vertical) in centered-Y-up space: D swaps to `(-y, -x)`, H negates x, V negates y. Point order is preserved — `Polygon.FromPoints` normalizes winding internally so the odd-flip-reverses-winding concern doesn't reach SAT. `TileShapeCollection.AddPolygonTileAtCell` now throws `InvalidOperationException` on a duplicate `(col, row)` instead of silently dropping the second polygon; multi-polygon-per-cell is unsupported and authors should merge shapes in Tiled.

## Slope Polygons in Tiled Maps — Phase 1
`TileMapCollisionGenerator` now inspects `TilemapTileData.CollisionObjects` when building collision from a tile layer. For each matched tile with one or more `TilemapPolygonObject`s (authored as `<polygon>` in a tileset tile's `<objectgroup>`), the generator converts points from Tiled pixel space (Y-down, origin top-left of tile) to local Polygon space (Y-up, centered on the cell: `localX = tiledX - G/2`, `localY = G/2 - tiledY`) and calls `TileShapeCollection.AddPolygonTileAtCell` with a prototype built from `Polygon.FromPoints`. Tiles without polygon objects still emit a rect via `AddTileAtCell`. Flip flags, rectangle/ellipse collision objects, and multiple polygons per cell are deferred to Phase 2 (see TODOS.md).

## TileNodeNetwork (A* Pathfinding)
`TileNodeNetwork` in `FlatRedBall2.AI` is a grid-based node network for A* pathfinding. Construct with `xOrigin`, `yOrigin`, `gridSpacing`, `xCount`, `yCount`, and `DirectionalType` (Four or Eight). Populate via `FillCompletely()` or `AddAndLinkNode(x, y)` (by tile index) / `AddAndLinkNodeAtWorld(wx, wy)`. Remove nodes via `RemoveAt(x, y)`, `RemoveAtWorld`, or `RemoveNodesOverlapping(AxisAlignedRectangle)`. For 8-way networks, call `EliminateCutCorners()` after carving walls to prevent diagonal movement through corners of missing tiles. Query with `NodeAt(x,y)`, `NodeAtWorld`, and `GetClosestNode`. Find paths with `GetPath(start, end)` (allocates) or `GetPath(start, end, List<Vector2>)` (reuse to avoid GC). Result contains world positions from start (exclusive) to end (inclusive). `TileNode.Tag` holds optional game data per node.

## ACHX Animation Format
`AnimationFrame`, `AnimationChain` (extends `List<AnimationFrame>`), and `AnimationChainList` (extends `List<AnimationChain>`, string indexer by name) are the runtime types in `FlatRedBall2.Animation`. `AnimationChainListSave` / `AnimationChainSave` / `AnimationFrameSave` in `FlatRedBall2.Animation.Content` handle XML deserialization of `.achx` files via `AnimationChainListSave.FromFile(path)` and conversion to runtime via `ToAnimationChainList(ContentManagerService)` (content pipeline) or `ToAnimationChainList(GraphicsDevice)` (raw file loading via `Texture2D.FromFile`; textures are cached per path). Supports `TimeMeasurementUnit` (Undefined/Second/Millisecond — Undefined treated as Second for FRB1 compatibility), `TextureCoordinateType` (UV/Pixel), and `FileRelativeTextures`. `AnimationFrame.RelativeX`/`RelativeY` are applied to the sprite's local `X`/`Y` on each frame switch so per-frame positional offsets work correctly. `Sprite.AnimationChains`, `PlayAnimation(string)`, `PlayAnimation(AnimationChain)`, `Animate`, `IsLooping`, `AnimationSpeed`, `CurrentAnimation`, and `AnimationFinished` event are the playback API. `Screen.Update` calls `Sprite.AnimateSelf` on all sprites each frame inside the `!IsPaused` block. `PlatformerSample` includes `AnimatedPlayer` — an entity with `PlatformerBehavior` and a `Sprite` driven by `PlatformerAnimations.achx` (FRBeefcake spritesheet), switching between Idle/Walk/Jump/Fall × Left/Right based on movement state each frame.

## Polygon — Concave Collision
`Polygon.ConvexParts` (`IReadOnlyList<IReadOnlyList<Vector2>>`) exposes the convex decomposition in local space. `FromPoints`, `SetPoints`, and `CreateRectangle` automatically decompose: convex polygons get a single part (the original points); concave polygons are ear-clip triangulated then Hertel-Mehlhorn merged into the minimum set of convex sub-polygons. `CollisionDispatcher` iterates `ConvexParts` in `PolygonVsPolygon`, `PolygonVsAabb`, and `PolygonVsCircle`, running SAT per part-pair and returning the minimum-magnitude MTV from any overlapping pairs. Shapes placed in concave pockets no longer produce false collisions.

## Spatial Partitioning (Sweep-and-Prune)
`Factory<T>.PartitionAxis` (type `Axis` — `X` or `Y`) opts a factory into broad-phase culling. The engine sorts each factory's list in-place once per frame (insertion sort — O(n) on nearly-sorted data) before collision relationships run. Any `CollisionRelationship` whose both lists are factories with matching non-null `PartitionAxis` automatically sweeps instead of doing O(n×m) checks — pairs separated by more than their combined `BroadPhaseRadius` are skipped. Mismatched or null axes silently fall back to full O(n×m). `CollisionRelationship.DeepCollisionCount` reports narrow-phase checks performed last frame for profiling. `TileShapeCollection` is already internally partitioned by grid lookup and does not need this.

`AudioManager` (at `Engine.Audio`) wraps MonoGame `SoundEffect` and `Song`. `Play(SoundEffect)` supports volume/pitch/pan, per-frame dedup (same sound plays at most once per frame), and a concurrent-instance cap (`MaxConcurrentSounds = 32`). Music via `PlaySong(Song, loop)` and `PlayPlaylist(params Song[])` with automatic playlist advancement. `SoundVolume`/`MusicVolume`/`SoundEnabled`/`MusicEnabled` for master control. Assets loaded via `Content.Load<SoundEffect>()` / `Content.Load<Song>()`.

## Per-Shape Collision Exclusion
`Add<T>(T child, bool isDefaultCollision)` attaches a shape without adding it to the default collision set. `SetDefaultCollision(ICollidable shape, bool)` toggles participation at runtime (idempotent; throws if shape is not a child of the entity). Demo in `DefaultCollisionDemoScreen`.

## Per-Shape Collision Targeting in Collision Relationships
`WithFirstShape(Func<A, ICollidable>)` and `WithSecondShape(Func<B, ICollidable>)` fluent methods on `CollisionRelationship<A, B>`. Selectors restrict which child shape is used for detection and response while keeping physics response applied to the parent entity. Demo in `ShapeSelectDemoScreen`.

## Y-Sort Rendering (Top-Down Draw Order)
`Screen.SortMode = SortMode.ZSecondaryParentY`. Items with different Z sort by Z; items with equal Z sort by parent entity's world-space Y descending. Uses insertion sort (stable, O(N) when nearly sorted).

`entity.MoveToLayer(layer)` updates the `Layer` property on all `IRenderable` children and Gum visual children, recursing into child entities. `IRenderable.Layer` has a public setter.

## Pause-Aware Delay
`TimeManager._sinceScreenStart` (exposed as `CurrentScreenTimeSeconds`) does not advance while `Screen.IsPaused`. `DelaySeconds` and `DelayUntil` compare against this value, so they automatically suspend during pause. `DelayFrames` is not pause-aware by design — frame counting continues regardless. `TimeScale` still applies to screen time. `DeltaSeconds` in `FrameTime` reflects real elapsed time (not zeroed when paused). `ResetScreen()` resets screen time on screen transition.

## Pause System
`Screen.IsPaused`, `PauseThisScreen()`, `UnpauseThisScreen()`. When paused, entity physics, collision, and entity `CustomActivity` are all skipped. `Screen.CustomActivity` always runs so the screen can respond to input (e.g., showing a pause menu). TopDownMenuSample updated to use an in-screen Gum overlay rather than a screen transition, so game world state is preserved across pause/resume. `BouncingBallsSample` added as a demo: click to spawn balls with gravity that bounce off a `TileShapeCollection` arena and self-collide; ESC toggles pause with a code-only Gum label.

## Tiled Tile Rendering via TilemapSpriteBatchRenderer
`TileMapLayerRenderable` delegates to MonoGame.Extended's `TilemapSpriteBatchRenderer.DrawLayer()` for per-layer rendering. This provides frustum culling (only visible tiles drawn) and automatic handling of all 8 tile flip/rotation combinations. A `TiledRenderBatch` (no-op `IRenderBatch`) ensures the `SpriteBatch` is not in an active state when the MGEx renderer manages its own Begin/End. FRB's Y-up camera is mapped to an `OrthographicCamera` in Tiled's Y-down space per draw call.

## BoundsRectangle Struct
`BoundsRectangle` (`readonly record struct` in `FlatRedBall2.Math`). Center-origin axis-aligned rectangle with `CenterX`, `CenterY`, `Width`, `Height` and computed `Left`/`Right`/`Top`/`Bottom` edges. Lightweight value type with no collision/rendering/attachment overhead — replaces `AxisAlignedRectangle` for non-collision uses like camera map bounds. Two-param constructor `BoundsRectangle(width, height)` centers at origin.
